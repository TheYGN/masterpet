## 0. מה אנחנו בכלל בודקים

הייבוא אמור לזהות שורות שטוחות שהן באמת וריאציות של אותו מוצר, ולקפל אותן ל:
- **מוצר-הורה אחד** (`products`) — שמו = הקידומת המשותפת (LCP) של השמות.
- **מספר variants** (`product_variants`) תחתיו — אחד לכל שורה.
- **מאפיינים ב-EAV** — הטעם הופך לערך של מאפיין **"טעם"**, וה-quantity+unit הופכים לערך של מאפיין **"גודל"**, מקושרים דרך `variant_attribute_values`.

**קריטריון ההצלחה העליון:** מוצר שיובא נראה למערכת **זהה** למוצר שנוצר ידנית — נפתח במסך עריכה עם הווריאציות, מסונן נכון, ומופיע נכון ב-KPI.

## 1. תנאי קדם — אל תדלג על זה (כאן נופלים)

### 1.1 מצב התחלתי נקי — מלכודת ה-SKU 🚨
**הבעיה:** מחיקת מוצר במערכת היא **soft delete** — היא מסמנת `products.deleted_at` בלבד. שורות ה-`product_variants` עם ה-SKUs **נשארות חיות ב-DB**, והאילוץ `UNIQUE(tenant_id, sku)` ממשיך לחסום. לכן ייבוא חוזר של אותו קובץ אחרי "מחיקה" ב-UI → כל השורות מדלגות (conflict=skip) → **שום דבר לא נוצר, וזה ייראה כאילו הבדיקה נכשלה כשבעצם היא בכלל לא רצה.**

**לפני כל ריצת בדיקה, ודא אחד מהשלושה:**
1. **SKUs ייחודיים** שלא קיימים עדיין ב-DB (הכי פשוט — תן לקובץ הבדיקה prefix כמו `QATEST-001`).
2. **מחיקה אמיתית (hard delete)** של שאריות קודמות דרך Supabase (לא דרך ה-UI). בגלל FK cascade, מחיקת ה-`products` מנקה גם variants/values/inventory:
   ```sql
   delete from products
   where tenant_id = '<TENANT_ID>'
     and (deleted_at is not null or name like 'QATEST%');
   ```
3. **Tenant / חנות בדיקה נקייה.**

> ⚠️ זהו גם **באג מוצרי ידוע** (לא קשור ל-normalization): מוצר מחוק חוסם ייבוא חוזר של אותו SKU בלי אינדיקציה למשתמש. תיקון עתידי = אינדקס unique חלקי (`WHERE deleted_at IS NULL`) + cascade של soft-delete ל-variants, או החייאה בייבוא. עד שזה יתוקן — הבדיקה חייבת להתחיל ממצב נקי.

### 1.2 מיפוי השדות החדשים
כדי לבדוק את מימד ה"גודל", חובה למפות בשלב המיפוי גם:
- **כמות מימד (גודל/משקל/נפח)** → `dimension.quantity`
- **יחידת מימד (גרם/מ"ל/ס"מ/יח׳)** → `dimension.unit`

אם לא ממפים אותם — הקיבוץ עדיין יעבוד לפי טעם, אבל לא ייווצר מאפיין "גודל", וגם **הקיבוץ לא יתפצל לפי יחידה** (כל המוצרים באותו דלי). זה תרחיש לגיטימי לבדיקה, אבל צריך לדעת מה מצפים.

### 1.3 אסטרטגיית קונפליקט
ברירת המחדל היא **skip**. לבדיקה ראשונית על מצב נקי — skip מתאים. אם בודקים merge/replace — צריך שכבר יהיו SKUs קיימים ב-DB.

## 2. עיצוב קובץ הבדיקה — חייב לכסות את כל המקרים

קובץ בדיקה "מושלם" **אינו** רשימה נקייה של משפחה אחת. הוא חייב לערבב מקרים, כי באג הקיבוץ האחרון (2026-05-29) התגלה דווקא כשכמה משפחות נפלו לאותו דלי. הקובץ חייב לכלול את כל הבאים:

| # | תרחיש | דוגמת שורות | תוצאה מצופה |
|---|---|---|---|
| A | **משפחה ברורה** — קידומת ≥2 מילים | `MONGE בוגר כבש`, `MONGE בוגר בקר`, `MONGE בוגר ארנבת` | הורה אחד "MONGE בוגר" + 3 variants (כבש/בקר/ארנבת) |
| B | **כמה משפחות באותה קטגוריה** | משפחה A + `MATCH בוגר עוף`, `MATCH בוגר דג` | 2 הורים נפרדים — "MONGE בוגר" ו-"MATCH בוגר". **זה התרחיש שתופס את באג הקיבוץ.** |
| C | **בודד אמיתי** — אין לו אח | `מסרק במבוק 69 שיניים` | מוצר בודד (grouped=false), בלי מאפיינים |
| D | **קידומת של מילה אחת בלבד** | `Royal Canin`, `Royal Treats` | 2 מוצרים בודדים — לא מתקבצים (safeguard ≥2 מילים) |
| E | **אותה משפחה, יחידות שונות** | `X בוגר עוף` (100 gram), `X בוגר עוף` (1 kg) | 2 מוצרים נפרדים — יחידה מפצלת דלי |
| F | **מלכודת אמצע-מילה** | `MONGE Tray Pate Beef`, `MONGE Tray Patatas` | הורה = "MONGE Tray" (לא "MONGE Tray Pat") |
| G | **עברית RTL + רווח קשיח/סימני כיווניות** | שמות עם NBSP / `‎`/`‏` באמצע | מתקבץ נכון (ה-tokenizer מנרמל אותם) |
| H | **מימד מלא** | משפחה עם quantity+unit שונים | כל variant מקבל מאפיין "טעם" ו"גודל" עם הערך הנכון |

**עמודות מינימום בקובץ:** `שם מוצר`, `SKU`, `מחיר`, `קטגוריה`, ואם בודקים מימד — `כמות` + `יחידה`. תן SKUs ייחודיים (`QATEST-###`).

> טיפ ל-Excel ישראלי: שמור כ-CSV UTF-8, או השתמש ב-XLSX. ודא שלפחות שורה אחת מכילה NBSP אמיתי כדי לכסות תרחיש G.

## 3. ריצת הבדיקה — שלב אחר שלב

1. **פתח את מסך מוצרים → "ייבוא מ-Excel".**
2. **העלאה** — גרור את קובץ הבדיקה. ודא שמספר השורות שזוהה תואם לקובץ.
3. **תצוגה** — בדוק שה-preview מציג את השורות הגולמיות. *(הערה: ה-preview עדיין מציג שטוח — הנרמול רץ server-side בלבד. זה follow-up ידוע, לא כשל.)*
4. **מיפוי** — מפה את כל העמודות, כולל `dimension.quantity` + `dimension.unit` (סעיף 1.2).
5. **אימות** — ודא שאין שורות אדומות (שגיאות). שורות עם SKU חסר → ייווצר SKU אוטומטי (זה משנה התנהגות קיבוץ? לא — הקיבוץ לפי שם, לא SKU).
6. **ייבוא** — לחץ ייבא. שים לב לדיווח: `imported` / `failed` + שגיאות.
7. עבור לטבלת המוצרים ואמת מול סעיף 4.

## 4. אימות תוצאות — UI + DB (שתי השכבות)

### 4.1 ב-UI
- כל **משפחה** (A, B, H) מופיעה כשורת מוצר **אחת** — לא כשורה לכל טעם.
- לחיצה על המוצר פותחת את ה-ProductSheet ומציגה את כל הווריאציות עם ה-attributes (טעם, גודל).
- בודדים (C, D, E) מופיעים כשורות נפרדות.
- ב-KPI: מספר המוצרים = מספר ההורים+הבודדים; מספר ה-VARIANTS = סך השורות בקובץ.

### 4.2 ב-DB (Supabase) — האמת האמיתית
החלף `<TENANT_ID>`:

```sql
-- (1) כמה variants לכל מוצר — משפחות צריכות >1, בודדים =1
select p.name, count(v.id) as variants
from products p
join product_variants v on v.product_id = p.id and v.deleted_at is null
where p.tenant_id = '<TENANT_ID>' and p.deleted_at is null
group by p.name order by variants desc;

-- (2) מאפיינים + ערכים שנוצרו לכל מוצר (צריך "טעם" ו/או "גודל" עם הערכים הנכונים)
select p.name, a.name as attribute, av.value
from products p
join product_attributes a on a.product_id = p.id
join product_attribute_values av on av.attribute_id = a.id
where p.tenant_id = '<TENANT_ID>' and p.deleted_at is null
order by p.name, a.position, av.position;

-- (3) קישור variant→ערך — כל variant צריך להיות מקושר לטעם/גודל הנכונים שלו
select p.name as product, v.sku, a.name as attribute, av.value
from product_variants v
join products p on p.id = v.product_id
join variant_attribute_values vav on vav.variant_id = v.id
join product_attribute_values av on av.id = vav.attribute_value_id
join product_attributes a on a.id = av.attribute_id
where p.tenant_id = '<TENANT_ID>' and p.deleted_at is null
order by p.name, v.sku;

-- (4) בדיקת "ערכים יתומים" — חייב להחזיר 0 שורות (אחרת ה-prune של H1 לא עבד)
select av.id, av.value
from product_attribute_values av
join product_attributes a on a.id = av.attribute_id
join products p on p.id = a.product_id
left join variant_attribute_values vav on vav.attribute_value_id = av.id
where p.tenant_id = '<TENANT_ID>' and p.deleted_at is null and vav.attribute_value_id is null;
```

**בדיקת loss-free:** `סך variants ב-DB == מספר שורות הקובץ` (פחות failed שדווחו). אם לא — שורות אבדו או הוכפלו.

## 5. רובריקת דירוג — למלא אחרי כל ריצה

לכל ממד תן ציון 0–5 (0 = נכשל לגמרי, 5 = מושלם), הכפל במשקל, וסכם ל-100.

| ממד | משקל | מה בודקים | ציון (0–5) | משוקלל |
|---|---|---|---|---|
| **קיבוץ נכון** | 30% | משפחות התקבצו, משפחות שונות לא התערבבו (תרחיש B), safeguard עבד (D) | | |
| **מיפוי מאפיינים** | 20% | טעם+גודל נכונים ב-EAV, קישורי variant נכונים (שאילתות 2+3) | | |
| **Loss-free** | 15% | סך variants = שורות; אין כפילויות; אין אובדן | | |
| **ניקיון / אין יתומים** | 15% | שאילתה 4 = 0 שורות; אין הורים יתומים; מצב התחלתי נקי | | |
| **קצוות וקונפליקטים** | 10% | NBSP (G), אמצע-מילה (F), יחידה מפצלת (E), SKU כפול בקובץ | | |
| **UX ודיווח** | 10% | ספירת imported/failed נכונה, שגיאות בעברית, אין קריסות | | |
| **סה"כ** | 100% | | | **/100** |

**ספי החלטה:**
- **≥ 90** → APPROVE. מוכן ל-commit.
- **75–89** → APPROVE-WITH-NITS. לתעד את החוסרים, אפשר לתקן ב-follow-up.
- **< 75** → CHANGES-REQUESTED. לתקן ולהריץ שוב.

## 6. הערכת מצב המשימה — נכון ל-2026-05-29 (כנה)

> זו הערכת ביצוע של עבודת ה-variant-normalization עצמה עד כה. נכתבת בכנות כדי ללמוד, לא כדי לייפות.

**מה בוצע טוב:**
- הלוגיקה הטהורה (LCP מבוסס-מילים, safeguard, loss-free) נכונה ומתועדת.
- אותרה ותוקנה בעיה ארכיטקטונית אמיתית: מעבר מ-JSONB ל-EAV הקיים, כך שמוצר מיובא וידני זהים למערכת. זו הייתה הכרעה נכונה.
- code-review רציני בשני סבבים תפס 4 באגים אמיתיים (כולל קריטי של SKU כפול תוך-קבוצה), וכולם תוקנו ואומתו.

**מה לא בוצע טוב — הלקח המרכזי:**
- **הבדיקות הראשונות השתמשו בנתונים "נקיים" מדי** — כל דלי = משפחה אחת. לכן באג הקיבוץ המהותי (LCP אחד לכל הדלי במקום קיבוץ תת-קבוצות) **לא נתפס לא ב-tests ולא ב-code-review**, והתגלה רק כשירין הריץ ייבוא אמיתי. הקוד "עבר" אבל לא עבד על נתוני אמת.
- **לא הופרד "passed review" מ-"verified on real data".** הוכרז כמעט-מוכן לפני בדיקת runtime על קובץ אמיתי.
- בנוסף, ההמלצה הראשונה ("תמחק ותייבא מחדש") הייתה שגויה — לא נלקח בחשבון ש-soft-delete לא משחרר SKU.

**מה לשפר (action items):**
1. **לבדוק תמיד על נתונים הטרוגניים** — כמה משפחות, ללא קטגוריה, עברית — *לפני* הכרזה על "done". סעיף 2 בקובץ הזה קיים בדיוק בשביל זה.
2. **לעולם לא להכריז "done" על feature שמשנה התנהגות UI/DB בלי qa-automation runtime** על המוצר האמיתי, לא רק unit-tests.
3. **לתקן את באג ה-platform** (soft-delete חוסם SKU) — נפרד מ-normalization, אבל חוסם בדיקות חוזרות.
4. **לחזור ל-code-reviewer** על שינוי הקיבוץ (clusterByPrefix) לפני commit — זהו שינוי בלב האלגוריתם שנכתב אחרי הביקורת.

**ציון נוכחי לאלגוריתם הקיבוץ:** ממתין לריצת QA אמיתית לפי הרובריקה למעלה. ה-unit-tests עוברים (כולל תרחיש B), אבל לא הוכח עדיין end-to-end על קובץ אמיתי + אימות DB.
